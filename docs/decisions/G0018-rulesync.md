## 背景

今の状況では、AIツールの進化が速すぎて、今日選んだ「正解」が数ヶ月後には変わっている可能性がある。

`rulesync` を導入して設定を共有する。

| 現状の課題 | `rulesync` による解決 | 導入後のメリット |
| :--- | :--- | :--- |
| 複数ツールで設定がバラバラ | プロンプトの一元管理 | どのAIでも同じ精度で開発可能 |
| 設定更新の手間がツール数分かかる | コマンドひとつで同期 | メンテナンスコストの大幅削減 |
| 特定ツールへの依存（ロックイン） | 設定のポータビリティ確保 | 常に最適なツールを選択できる |

### 1. AIツールの「マルチプラットフォーム化」への対応

現在、Claude Code、Gemini CLI、GitHub Copilot CLIなど、強力なツールが乱立している。

- **ツールの使い分け:** 「ロジック構築はClaude、リサーチや大量のコンテキスト読み込みはGemini」といった具合に、目的別で使い分けるのが一般的。
- **設定の断片化:** それぞれのツールが独自の命令セット（`.clauderc` や各種設定ファイル）を必要とするため、ルールを更新するたびに全てのファイルを書き換えるのは非現実的。

### 2. 命令（プロンプト）の同期と整合性の維持

開発ルールやコーディング規約（例：「常にTypeScriptの型定義を厳格にする」「テストはVitestで書く」）は、プロジェクトごとに共通であるべき。

- **一貫性の欠如:** あるツールには最新のルールを教えたが、別のツールには古い指示が残っていると、AIの回答に食い違いが生じ、かえって手戻りが増える。
- **Single Source of Truth:** `rulesync` を導入することで、**「ひとつのマスター設定」**を各ツールの設定ファイルへ自動反映させ、どのAIを使っても同じ規約に従わせることが可能。

### 3. ベンダーロックインの回避と柔軟性

特定のCLIツールに依存しすぎると、そのツールが課金体系を変えたり、開発が止まったりした際に、積み上げた「秘伝のプロンプト」が資産として死んでしまう。

- **抽象化層としての機能:** `rulesync` はいわば、AIに対する指示の抽象化層。
- **即時切り替え:** 新しい画期的なAIツールが登場しても、`rulesync` の同期対象に加えるだけで、これまでの開発ルールをそのまま引き継いで使い始めることができる。

## 想定される影響

rulesyncのメンテが面倒。「最大公約数」的な設定をしないといけないためあれこれ考えることが多い。
