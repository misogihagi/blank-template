## 決定

GitHub Actionsのワークフローファイルにおける構文エラー、タイポ、およびセキュリティ上のベストプラクティスを自動的にチェックするため、actionlintを静的解析ツールとして導入する。

## 背景

* **状況**: GitHub Actionsを利用したCI/CDパイプラインが複雑になる恐れがある
* **懸念事項**: 実行するまでエラーが判明せず、デバッグのために「修正コミット」を何度もプッシュする必要があり、開発効率が低下している。また、`${{ github.event.inputs... }}` のような式におけるインジェクション脆弱性のリスクがある。
* **品質**: ワークフローの信頼性とセキュリティを向上させ、レビューの負担を軽減したい。
* **欠点**: 厳格なチェックにより、既存の（動いているが推奨されない）記述に対して修正コストが発生する。
* **決定**: 専用のリントツールであるactionlintを導入し、CIプロセスに組み込む。

## 想定される影響

* **簡単になること**:
  * 実行前に構文ミスや型の不一致を特定できる。
  * セキュリティリスク（シェルインジェクションなど）の早期発見。
  * レビュー時の際、ロジックに集中できる（単純な記述ミスを指摘しなくて済む）。
* **難しくなること**:
  * 複雑な式やカスタムアクションのメタデータ定義において、リントを通すための微調整が必要になる場合がある。
  * ローカル環境へのバイナリ（またはDocker経由）のセットアップが必要。

## 検討した他の選択肢

* **yamllint**: YAMLとしての構文はチェックできるが、GitHub Actions特有のコンテキスト（`matrix`や`steps`の依存関係、式言語など）を解釈できないため不十分。
* **GitHub公式のバリデーター (JSON Schema)**: エディタ上での補完には役立つが、複雑なセキュリティチェックやスクリプト内のリントまではカバーできない。
* **導入しない**: 実行エラーを繰り返すことでコストが増大し続けるため、非効率と判断。
