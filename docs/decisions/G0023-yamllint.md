## 決定

プロジェクト内のすべてのYAMLファイルに対して、**yamllint** を導入し、CI/CDパイプライン（GitHub Actionsなど）での自動チェックを強制する。

## 背景

* **ユースケース:** Infrastructure as Code (Ansible, CloudFormation, Kubernetes等) や CI/CD設定、各種ツールでYAMLファイルが多用される状況において、
* **懸念事項:** 記述スタイルのばらつき（インデント、改行、クォートの有無など）や、構文ミスによるデプロイエラーの発生に直面し、
* **品質:** 設定ファイルの可読性、保守性、および構文の正確性を達成するために、
* **欠点:** 既存のルールに準拠していないファイルの修正コストと、Lintエラーによるビルドの一時的な停止を受け入れ、
* **オプション:** グローバルなデファクトスタンダードである `yamllint` を導入することを決定した。

## 想定される影響

* **簡単になること:**
  * コードレビューにおける「インデントのズレ」などの些細な指摘が不要になる。
  * 構文エラーが実行前に検知されるため、デプロイの失敗が減少する。
  * プロジェクト全体で一貫したスタイルが維持される。
* **難しくなること:**
  * 初回導入時に、既存の大量の警告・エラーを修正する工数が発生する。
  * 厳格なルール設定にした場合、記述時にわずかなスタイルの差でもエラーとなり、開発のリズムが一時的に阻害される可能性がある。

## 検討した他の選択肢

* **Prettier (YAML plugin):**
  * フォーマッタとしては優秀だが、
  * 論理的なエラー（重複キーなど）の検出においては
  * `yamllint` の方が、詳細な設定が可能であるため、今回は静的解析に特化した `yamllint` を優先した。
* **手動レビューのみ:**
  * ヒューマンエラーを完全に排除できず、レビューコストが増大し続けるため却下。
