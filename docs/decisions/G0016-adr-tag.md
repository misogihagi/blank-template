## 決定

タグ（識別子）を導入し、名前空間を分離させる。

具体的には通し番号の前にプレフィックスを入れる。
プレフィックス（タグ）は以下の通り。

| 種別 | 役割・スコープ | 主な決定内容の例 |
| :--- | :--- | :--- |
| **G (Governance, General)** | 全体に影響がある決定 | ドキュメント、セキュリティポリシー、CI/CD方針、品質管理 |
| **F (Framework, Frontend)** | 特定の技術に依存した決定 | ツールの設定、拡張機能 |
| **P (Product)** | 事業要件に伴う構造決定 | プロダクト仕様や使ってみて起きたフィードバック |

## 背景

以前は、ADRは全てフラットで、単純な連番（001, 002...）でやっていた。
ただし、それだと同じタイミングでADRが書かれた際に同じ番号が振られるため、番号がコンフリクトを起こしていた。

### 1. 物理的な衝突（コンフリクト）の回避
もっとも直接的な理由は、**並行開発における作業のブロッキングを防ぐため**です。

* **中央集権的な管理の限界:** 全員が同じ `adr/` フォルダ内の連番を使っていると、チームAが `005` を作ろうとしている間に、チームBが先に `005` をマージしてしまうと、チームAは番号を振り直してリベース（修正）する手間が発生します。
* **分散開発への対応:** `FE-001` と `BE-001` のように名前空間を分ければ、フロントエンドとバックエンドのチームはお互いの進捗を気にせず、自分たちのコンテキスト内で意思決定を完結できます。


### 2. 認知的負荷（コンテキスト）の分離
人間が一度に把握できる情報の量には限界があります。

* **情報のノイズ除去:** インフラ担当者にとって、UIコンポーネントのライブラリ選定（ADR）は、日常業務においては「ノイズ」になりがちです。タグやディレクトリで分離されていないと、全てのADRをスキャンして自分に関係あるものを探さなければなりません。
* **メンタルモデルの整理:** 「これはアプリケーション層の決定」「これはインフラ層の決定」と境界線（バウンダリ）が明確になっていることで、エンジニアは必要な情報に素早くアクセスでき、脳のメモリを節約できます。

### 3. 検索性とトレーサビリティの向上
決定事項が増え、100、200と積み重なった時に、過去の経緯を追うのが困難になります。

* **ドメインごとの履歴管理:** 特定のサブシステム（例：決済基盤）に関するADRだけを抽出したい場合、タグがあれば `grep` や検索ツールで一瞬でフィルタリングできます。
* **依存関係の可視化:** 「なぜこのAPIはこの形なのか？」を知りたいとき、`API-` タグがついたADRだけを時系列で追えば、その設計思想の変遷がクリアに見えてきます。

## 想定される影響

### 1. ポジティブな影響（運用の円滑化）

#### **マージコンフリクトの激減**
もっとも直接的な効果です。`FE-001` と `BE-001` が並行して作成されても、ファイル名が衝突しないため、プルリクエスト（PR）の段階で「番号が被ったから修正して」という不毛なやり取りが消えます。

#### **オンボーディング（新参入者）の負担軽減**
新しいメンバーが入ってきた際、「全部で100個あるADRを全部読んで」と言うのは酷です。
* **影響:** 「まずは `FE-`（フロントエンド）のADRだけ目を通しておいて」と指示を具体化できるため、新人が業務に必要なコンテキストをキャッチアップするスピードが上がります。

#### **意思決定のオーナーシップの明確化**
タグをつける行為は「これは誰（どのチーム）が決めるべき領域か」を定義することに繋がります。
* **影響:** 「バックエンドのタグが付いているから、バックエンドチームの承認が必要だね」と、レビューのフローが自然と整理されます。

### 2. 注意すべき影響（トレードオフと対策）

#### **「どのタグにするか」迷うコストの発生**
境界線が曖昧な決定事項（例：認証周りはFEかBEか？）が出てきた際、どこに書くべきか迷って手が止まることがあります。
* **影響:** 「タグを決めるのが面倒だからADRを書くのをやめよう」という本末転倒な心理的ハードルが生じます。
* **対策:** **迷ったらタグは振らなくてもOK**、または、**G(汎用)を使う**。
