## 決定

プロジェクト内のMarkdownファイルの品質と一貫性を保証するため、校正ツールとして`textlint`を導入する。

## 背景

**ドキュメント作成・更新** の状況において、**表記揺れや誤字脱字が発生し、気づかないうちに混入しているという課題** に直面し、**ドキュメントの可読性と信頼性（品質）** を達成するために、**環境構築の手間やCI実行時間のわずかな増加（欠点）** を受け入れ、**静的解析ツールによる自動校正（textlint）の導入** を決定した。

プロジェクト内のドキュメントやREADMEファイルはMarkdown形式で記述されている。これらのMarkdownファイルに一貫性がなく、フォーマットのばらつきが見られることがあった。これにより、可読性が低下し、レビュープロセスでの手動での指摘が増えるという問題が発生していた。Markdownファイルの品質を自動的に保証し、開発者がフォーマットに関する手動での修正に時間を費やすことを減らす必要がある。

## 想定される影響

* **簡単になること**
  * 送り仮名のミス、二重否定、表記揺れ（例：「サーバ」と「サーバー」）の自動検出。
  * プルリクエスト（PR）における、本質的ではない「てにをは」の指摘作業の削減。
  * 執筆者によるセルフチェックの効率化。
* **難しくなること**
  * ルールの誤検知（False Positive）が発生した際、個別に除外設定（\`\`など）を記述する手間。
  * プロジェクト初期における、既存の大量の指摘事項への修正コスト。

## 検討した他の選択肢

* **RedPen**: 高機能だが Java 実行環境が必要であり、Node.js ベースのフロントエンド/バックエンドプロジェクトにおいては textlint の方が導入・共有が容易と判断。
* **人力レビューのみ**: 属人性が高く、微細なミスの見落としが避けられないため、自動化による補助が不可欠と判断。
* **VS Code 拡張機能（個人任せ）**: チーム全体で一貫したルールを強制できないため、設定ファイルを共有できるツールとしての導入を優先。
